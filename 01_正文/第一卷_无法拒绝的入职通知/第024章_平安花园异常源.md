# 第024章 平安花园异常源

公告栏上的纸很新。

新得不像这个小区的东西。

平安花园的公告栏一直带着一股旧胶水、雨水和灰尘混在一起的味道。

可这张【业主共同确认表】白得刺眼。

像刚从某个没有时间的打印机里吐出来。

第一行：

【请门卫确认：平安花园是否存在二号楼。】

第二行：

【请门卫确认：二号楼404是否为有效住户空间。】

第三行：

【请门卫确认：当前待核人员是否可按住户、维修、员工或访客分类处理。】

凉生没有走出门卫室。

他用监控放大公告栏。

只记录画面。

不确认纸张实体。

【01:31，公告栏监控画面出现“业主共同确认表”。本人未离开门卫室，未触碰原件，仅记录监控可见内容。】

工作手机弹出：

【共同确认表属于小区业主自治材料。】

【请配合确认。】

业主自治。

这词比红章还会装。

公司至少还承认自己是公司。

业主共同确认表一出来，像是所有住户都在背后点头。

王德福。

无脸小女孩。

红裙女人。

二号楼404。

维修工。

甚至门外那个辅助执行人。

只要被放进“共同确认”，它们就不再是单个异常。

它们会变成小区事实。

凉生问工作手机：“该表是否有业主签名？”

【业主共同意愿已生成。】

“是否可显示签名？”

【不适用。】

“是否有物业、居委会或门禁系统登记编号？”

【不适用。】

“不适用是什么意思？”

【共同确认优先于单项签名。】

凉生笑了。

“你们连群聊投票都比这讲规矩。”

手机没理他。

门卫室外的喇叭忽然响了。

老旧的小区广播发出滋啦声。

随后，一个温和的男声说：

“为了提高平安花园住户服务效率，请门卫室尽快确认二号楼相关事项。”

“长期待核会影响住户满意度。”

“住户满意度过低，将影响夜班保安岗位稳定性。”

住户满意度。

凉生想起王德福那张投诉单。

可信度被压到 0 的时候，也是这个东西先伸手。

平安花园不是单靠门禁吃人。

它靠确认。

住户投诉确认保安可信度。

名单确认住户身份。

监控确认楼栋。

维修单确认材料。

员工牌确认岗位。

现在，业主共同确认表要把所有零散确认拉成一条链。

谁站在门卫室，谁就被要求给这条链盖最后一个口头章。

凉生低头，在复核页上写：

【异常源初步定位：平安花园通过“确认链”将监控、门禁、名单、维修、员工牌及业主意愿连接为小区事实。门卫确认可能是链条末端动作。】

工作手机没有报警。

因为他写了“初步”。

没有写结论。

门外辅助执行人开口：

“确认二号楼存在，可以一次性解决空间待核。”

凉生说：“也可以一次性制造空间。”

“长期待核会降低满意度。”

“你又不怕扣分。”

辅助执行人沉默。

公告栏上共同确认表第三行开始闪。

【请门卫确认：当前待核人员是否可按住户、维修、员工或访客分类处理。】

门禁屏同时亮起五个申请。

【王德福：住户外出返回。】

【赵瑞：维修材料入库。】

【LS-0698：岗位交接完成。】

【露雨：访客改错机会使用。】

【二号楼404：空间登记补全。】

五项同时出现。

整齐。

漂亮。

像终于有人把杂乱的待核事项整理好。

凉生看着它们，心里只有一个判断。

太整齐了。

这不是复核。

这是清仓。

他在旧部门复核页新增三栏。

【执行人】

【确认人】

【承担人】

然后逐项填写。

王德福。

执行人：门卫。

确认人：要求为门卫。

承担人：若放行错误，住户名单、可信度和入口门责任落到夜班保安。

赵瑞。

执行人：维修流程。

确认人：要求为门卫。

承担人：若材料化错误，候选人残留姓名被转为小区维修材料，后果未知，当前不能确认由维修系统承担。

LS-0698。

执行人：岗位交接流程。

确认人：要求为门卫。

承担人：若交接错误，LS-0713 可能承接残留岗位资源和离职状态冲突。

露雨。

执行人：门禁访客流程。

确认人：要求为门卫。

承担人：若借名错误，真实露雨记录可能被污染，改错机会来源被覆盖。

二号楼404。

执行人：空间登记流程。

确认人：要求为门卫。

承担人：若补全错误，监控空间可能获得现实入口，门卫成为路径确认人。

五项写完，凉生在最后一行写：

【共同确认表将五类不同后果压缩为“住户服务效率”问题，存在异常确认链风险。】

工作手机红光大亮。

【员工 LS-0713 正在质疑业主共同意愿。】

凉生补：

【本人不质疑业主共同意愿，只记录当前未见具体业主签名、授权编号和后果承担人。】

红光停了。

公告栏却没有停。

那张纸的底部慢慢浮出一排名字。

王德福。

王德福孙女。

二号楼404住户。

维修人员。

夜班保安 LS-0713。

最后一个名字让凉生眼神沉下来。

共同确认表把他也写进了业主共同意愿。

门外辅助执行人轻声说：

“你已经在表里。”

凉生看着公告栏。

“表里写我，不等于我写表。”

他立刻记录：

【共同确认表出现“夜班保安 LS-0713”字样。本人未参与填写，未签名，未授权其代表本人意愿。】

工作手机提示：

【检测到本人意愿争议。】

【请确认是否撤销 LS-0713 在共同确认表中的默认参与。】

这个问题看起来终于对了。

但凉生没有点。

撤销默认参与。

听着像保护自己。

可如果他只撤销自己，那其他名字就默认成立了。

王德福孙女。

二号楼404住户。

维修人员。

它们都会被保留下来。

凉生问：“是否可只记录本人未参与，而不对其他名字有效性作判断？”

【可记录。】

“是否可要求逐项出示签名或授权来源？”

【低权限员工无权要求全体业主出示签名。】

“是否可要求仅用于本次入口门动作？”

手机停顿。

【可限定当前岗位动作。】

够了。

凉生写：

【本人仅确认：LS-0713 未参与共同确认表填写。对表中其他名称不作真实性判断。该表不得直接用于当前入口门放行、空间登记、岗位交接或材料转化，仅可作为待核材料进入证据夹。】

公告栏上的纸抖了一下。

像有人在背后猛地拽住它。

小区广播再次响起。

这次声音变冷。

“门卫拒绝服务住户。”

“门卫拖延维修。”

“门卫影响小区正常秩序。”

“门卫满意度持续下降。”

工作手机弹出：

【住户满意度：-37】

【保安可信度：1 → 0】

凉生看着数字归零。

没有慌。

第006章他已经见过一次。

可信度归零不是死刑。

它只是逼你证明门卫室里还有一个能接收投诉的人。

这次不能机械重复。

因为投诉已经不是王德福一个人。

是整个小区。

凉生翻出所有记录。

访客登记。

值班日志。

门禁原始记录。

维修单。

旧部门复核页。

辅助执行边界记录。

他在新页写：

【共同确认表既然要求门卫确认，说明门卫岗位仍被该表视为有效确认节点。若门卫岗位完全失去可信度，则共同确认表自身无法要求门卫完成确认。】

工作手机闪了一下。

没有报警。

凉生继续写：

【因此，当前至少存在一项功能性可信度：接收确认请求并记录不能确认的理由。】

【保安可信度不恢复为服务满意，仅恢复为确认节点最低有效。】

手机弹出：

【保安可信度：0 → 1】

【类型：确认节点最低有效。】

凉生看着这个“1”。

还是那个穷酸数字。

但能用。

公告栏上的【业主共同确认表】边缘开始发黑。

它没有消失。

只是从白纸变成了半透明。

纸后面露出一个旧东西。

不是公告。

是一台嵌在墙里的铁箱。

铁箱上贴着褪色标签。

【住户意见箱】

箱口很窄。

里面黑得看不见底。

工作手机弹出：

【平安花园异常源已显露。】

【名称：住户意见箱。】

【功能：收集住户确认，生成小区共同事实。】

【请员工 LS-0713 进行处理。】

门外辅助执行人看着那只箱子。

第一次没有开口。

凉生也没有马上处理。

他只写：

【异常源显露不等于异常源已解释。当前只确认住户意见箱与共同确认表存在关联。】

意见箱的箱口忽然吐出一张纸条。

纸条飘到门卫室玻璃外。

上面写着一行字：

【请确认你愿意成为本小区住户。】

下面已经预填好名字。

【凉生】
