# 小说项目长期工作规则

## 一、规则适用范围

-   本文件是整个小说项目的长期默认规则。
-   用户当前任务中的明确指令优先于本文件。
-   如果用户指令与本文件存在明显冲突，先指出冲突，再按照用户确认后的要求执行。
-   不得因为推测用户意图而擅自扩大任务范围。

## 二、项目基本信息

-   小说名称为《这份工作，活着才能转正》。
-   全部小说正文、框架、设定、伏笔和修改说明使用简体中文。
-   正文、大纲、设定、伏笔和章节规划统一使用 Markdown 格式。
-   男主固定为凉生。
-   女主固定为露雨。
-   本书为单女主、无后宫。
-   不新增长期暧昧对象、隐藏女主、替代女主或与露雨同等地位的感情角色。
-   除已经发布的正文外，`02_大纲与设定/01_全书总体大纲.md`
    是所有未发布剧情的最高依据。

## 三、剧情核心限制

-   凉生主要依靠观察、试探、验证、推理和利用规则冲突解决问题。
-   不使用血统觉醒、天选救世主、前世神明身份、突然恢复前世力量等设定。
-   不使用系统无条件赠送能力、道具、属性或奖励的情节。
-   凉生可以失败、误判和付出代价，不能依靠作者强行安排脱险。
-   露雨必须拥有独立目标、主动行动、判断和完整成长。
-   露雨不能只承担提示规则、解释世界观、提供线索或等待救援的功能。
-   后期必须继续保留合同、岗位、权限、绩效、审核、申诉、调岗、晋升、降职和裁员等职场元素。
-   不得把故事改成普通玄幻、修仙、末世刷怪或无限副本题材。
-   不得为了制造爽点，临时增加未铺垫的新能力、新规则或万能道具。

## 四、正文创作规则

-   正文必须延续相邻章节已经形成的叙事视角、人物语气和语言风格。
-   不得在没有用户要求的情况下改变叙事视角或写作风格。
-   正文不能写成剧情大纲、设定说明或事件流水账。
-   新规则应通过人物行动、观察、试错和后果逐步展现，避免集中解释。
-   人物只能依据自己已经获得的信息作出判断，不能突然知道尚未发现的事实。
-   新增人物、组织、能力、岗位或规则前，必须确认总体大纲和人物设定中是否已有依据。
-   临时场景人物可以出现，但不得承担长期设定、核心规则、感情线或关键解谜功能；若临时人物转为长期角色，必须补足大纲或人物设定依据。
-   每章必须具有明确的当前目标、实际阻碍、剧情变化和后续推动。
-   章节结尾可以保留悬念，但不得每章机械重复同一种反转方式。
-   前后文衔接时，下一章开篇不得机械重复上一章片尾已经写过的画面、对白、系统提示或同一段剧情，只能承接其结果、后果或新的变化。
-   章节片尾不得预演下一章开篇将要原样重复的剧情；连续章节的开篇和片尾应形成推进、转向或新增压力，避免同一悬念、同一提示、同一敲门、同一弹窗或同一待核方式反复收尾。
-   每次续写小说前，必须先检查 `01_正文`
    中已有正文源文件，以现有最高章节及相邻正文的实际内容作为直接续写起点；不得只依据已发布章节进度跳过、重写或覆盖已经存在的未发布正文稿。
-   默认“继续写”是从 `01_正文` 实际最高章节继续；如果用户明确要求“按发布进度继续”或“重写未发布正文”，则从当前已发布后一章开始处理，并只改未发布章节。
-   已发布正文用于确定不可冲突的正史保护边界；未发布正文若与已发布正文或核心设定冲突，应先说明并按文件优先级处理，不得静默续写。
-   正文创作和剧情修改必须检查 `02_大纲与设定/05_后续写作风险.md`
    中与已发布正文绑定的风险。
-   后续写作风险不要求在固定章节或固定时间内立刻解决，可以通过后续剧情推进、伏笔转化、伏笔回收或设定解释逐步处理。
-   写作风险转化为伏笔、长期线索或后续谜题时，必须兼容已发布正文和总体框架，不得偏离全书主线、人物设定、章节规划和伏笔时间线。
-   不得为了消除写作风险，临时增加未铺垫设定、强行否认已发布正文、制造人物信息越权、时间线断裂或规则逻辑问题。
-   每章篇幅、标题和具体情节以用户本次指令为准。
-   用户未要求续写多章时，只能完成指定章节，不得自动继续下一章。
-   写作流程应尽量简化：先确认 `01_正文`
    实际最新章节和本次承接点，再读取本次直接相关设定、相邻正文、未完成伏笔和未解除风险，随后直接写正文并同步必要归档。
-   除非用户要求详细分析或存在设定冲突，不得在开始写作前输出冗长的逐项确认、长篇计划或完整设定复述。

## 五、文风、幽默、去 AI 化与爽点规则

### 1. 整体文风

-   小说整体保持紧张、悬疑、诡异和职场规则怪谈的主基调。
-   在不破坏危险感和悬念的前提下，适度加入自然幽默，用于调节阅读节奏和增强人物真实感。
-   幽默必须来自人物性格、心理活动、对话反差、现实反应和荒诞处境，不得为了搞笑强行插入网络段子、流行梗或与世界观无关的笑话。
-   凉生的幽默以冷静吐槽、一本正经和反差感为主，不得写成话痨、段子手或纯搞笑角色。
-   危险场景中的幽默只能短暂缓解阅读压力，不得削弱死亡威胁、诡异感和剧情紧迫感。
-   不得为了增加幽默、爽点或可读性，把作品改成轻喜剧、无脑爽文或普通升级流。

### 2. 去 AI 化要求

-   避免过度工整、对称、总结式和模板化表达。
-   避免频繁使用"不是......而是......""这一刻，他终于明白......""真正可怕的从来不是......"等常见模板句式。
-   减少无意义环境描写、重复心理描写、同义反复和已经通过行动表现后又再次解释的内容。
-   人物对话允许停顿、打断、答非所问、说半句话和不完整表达，使交流更接近真实人物。
-   不同人物必须具有不同的语言习惯、信息边界和思考方式，不得所有人物都说话完整、理性且风格相同。
-   优先通过动作、细节、选择和后果表现情绪，减少直接说明人物"很紧张""很害怕""很震惊"。
-   句子长短和段落节奏应自然变化，不得连续使用结构高度相似的句子和段落。
-   不追求每句话都是金句，允许保留符合人物和场景的自然表达。
-   不得在每段剧情结束后总结意义、复述结论或替读者解释已经能够理解的内容。
-   正文完成后必须进行一次去 AI
    化检查，重点删除模板句、机械排比、重复解释、过度总结、空泛抒情和不符合人物身份的精致台词。

### 3. 复杂逻辑的解释方式

遇到复杂规则、推理链、时间线、多人关系、权限关系或规则冲突时，按照以下顺序处理：

1.  先通过剧情展示问题，让读者产生疑问。
2.  再由凉生通过观察、试探、验证、失败或与其他人物交流逐步解释。
3.  凉生不得直接获得答案，关键结论必须有前文细节或实际验证作为依据。
4.  优先使用人物能自然说出口的话解释，不得让正文变成作者讲课或设定说明书。
5.  只有在人物解释后仍然存在较高理解门槛时，才允许加入简洁的辅助图示。

### 4. 思维导图使用规则

-   思维导图只用于真正复杂的规则关系、推理链、时间线、多人关系或规则冲突，不得频繁出现。
-   必须先有凉生在剧情中的观察、推演或解释，再考虑加入思维导图。
-   思维导图只能辅助理解，不能代替剧情、人物行动和推理过程。
-   简单逻辑禁止使用思维导图。
-   优先使用占用空间较小的 Mermaid `mindmap`、`flowchart` 或 Markdown
    树状结构。
-   图中只保留关键节点，不得把正文内容重新复制一遍。
-   思维导图默认只用于作者、Codex
    和用户理解复杂逻辑，不直接写入正式小说正文或发布稿。
-   只有用户明确要求，或图示本身属于剧情内合理出现的文件、记录、规则材料时，才允许进入正文或发布稿。
-   如果允许进入发布稿且发布平台不支持
    Mermaid，发布前必须转换为纯文本树状结构或其他平台可正常显示的简洁形式。
-   思维导图不得出现在会明显打断追逐、战斗、逃亡、惊吓或其他高紧张场景的位置，应在阶段性停顿、复盘或解释节点使用。

### 5. 爽点与阅读满足感

-   在不偏离紧张、悬疑、诡异和职场规则怪谈主调的前提下，适度增加爽点和阶段性阅读满足感。
-   爽点必须来自已有剧情、规则、人物能力、信息差和前期铺垫，不得突然开挂。
-   凉生的核心爽感来自观察、验证、提前留后手、利用信息差和规则冲突完成反制。
-   凉生可以赢，但胜利不能让世界失去危险感；重要胜利应保留合理代价、限制或新的问题。
-   当剧情长期处于压抑、受挫或高强度推理状态时，应根据剧情自然寻找阶段性回报机会；并非每次危机后都必须安排爽点，也不得为了满足节奏要求强行制造胜利或提前泄露谜底。

优先使用以下爽点：

-   **规则反杀**：敌人利用规则逼迫凉生，凉生发现漏洞、隐藏条件或规则冲突，再反过来限制对方。
-   **信息差反制**：他人以为凉生不知道某件事，关键时刻才发现凉生已经通过细节察觉并提前准备。
-   **验证成功**：凉生提出危险或反常的猜测，通过低成本试探获得证据，并因此避开更大危险。
-   **绝境翻盘**：解决办法必须有前文线索，翻盘后读者回看前文能够发现依据。
-   **借力破局**：利用规则之间的冲突、不同势力的矛盾或敌人的误判完成反制。
-   **认知领先**：众人仍在处理表面问题时，凉生已经发现真正的问题，但其他人物不得因此被强行降智。
-   **成果兑现**：前期积累的人情、线索、道具、知识、承诺和伏笔在后续产生实际作用。

### 6. 爽点节奏与限制

-   不要求每章都有大爽点，但不得长时间只有压抑、受挫和谜团而没有任何回报。
-   小爽点可以来自判断正确、识破谎言、提前留后手或让对手付出代价。
-   中型爽点可以来自一次完整破局、救下关键人物或破解重要规则。
-   大爽点应主要留给卷内高潮、重大伏笔兑现和核心敌人反制。
-   禁止突然觉醒隐藏血统、无铺垫能力或万能道具。
-   禁止系统无条件发放刚好能解决当前危机的奖励。
-   禁止为了突出凉生而让其他人物集体降智。
-   禁止反派明明可以行动，却停下来听凉生完整解释全部计划。
-   禁止依靠重复打脸、围观群众震惊和机械嘲讽制造廉价爽感。
-   禁止让凉生每次都算无遗策或连续轻松获胜。

以下链条可作为凉生典型破局方式之一，不要求完整使用，也不得连续多次机械套用同一结构。应根据当前剧情删减、打乱或变化其中步骤：

观察异常\
→ 提出暂时无法确认的猜测\
→ 低成本验证\
→ 保留部分信息或后手\
→ 危机真正出现\
→ 对手认为凉生无路可走\
→ 前期细节开始兑现\
→ 利用规则冲突或信息差完成反制\
→ 付出合理代价\
→ 获得阶段性胜利\
→ 留下新的、更大的问题

### 7. 最高原则

-   幽默服务于人物，不能破坏危险感。
-   解释服务于剧情，不能把正文写成说明书。
-   思维导图服务于理解，不能代替人物推理。
-   爽点服务于阅读满足感，不能改变作品主调和核心逻辑。
-   读者可以因为凉生破局而感到爽，但不能因此失去对这个世界的恐惧。
-   每一次胜利都应该证明凉生更会活下去，而不是证明这个世界已经拿他没办法。

## 六、开始任务前的读取要求

进行正文创作或剧情修改前，必须根据任务读取以下文件：

1.  根目录 `AGENTS.md`。
2.  `02_大纲与设定/01_全书总体大纲.md`。
3.  `02_大纲与设定/02_人物与世界观.md`。
4.  `02_大纲与设定/03_分卷与章节规划.md`。
5.  `02_大纲与设定/04_伏笔与时间线.md`。
6.  `02_大纲与设定/05_后续写作风险.md`。
7.  `01_正文`
    中当前实际最高章节、当前任务涉及的相邻正文和相关已发布正文。

读取文档时遵循以下精简规则：

-   `04_伏笔与时间线.md`
    中状态已经明确为已回收、已完成、已解除、已修正，且与本次章节无直接关系的伏笔，只需确认编号和状态，不必反复细读全文。
-   `05_后续写作风险.md`
    中状态已经明确为已解除、已修正，且不会影响本次承接点的错误或风险，只需确认状态，不必纳入本次重点检查。
-   已完成伏笔、已解除风险或已修正错误如果涉及当前章节、相邻正文、已发布正文保护、人物信息边界或本次要使用的规则，仍必须读取对应条目，避免误写。
-   优先读取未完成、持续推进、近期重点、待写入正文、待复核、计划中且与本章有关的伏笔、风险和章节规划。

读取后必须先确认：

-   `01_正文`
    中实际写到的最新章节，以及本次应从哪一章、哪一个剧情尾点继续；
-   当前章节所处分卷和时间点；
-   凉生与露雨当前掌握的信息；
-   当前已经出现的规则、道具和人物；
-   本章需要推进或回收的伏笔；
-   本章涉及的已发布正文绑定风险，以及本章是否处理、暂缓或转化为伏笔；
-   本章计划使用的幽默、爽点和复杂逻辑解释是否符合主调；
-   如果存在复杂逻辑，是否真的需要思维导图辅助；
-   是否存在与已发布正文冲突的内容。

如果核心文档之间存在无法按照文件优先级解决的矛盾，停止修改并向用户说明。

## 七、文件管理规则

-   正文源文件只放在 `01_正文`。

-   以下正史内容只维护在 `02_大纲与设定` 的四份核心文档中：

    -   `01_全书总体大纲.md`：全书主线、分卷目标、核心结局。
    -   `02_人物与世界观.md`：人物档案、关系、世界规则和组织设定。
    -   `03_分卷与章节规划.md`：各卷及各章具体规划。
    -   `04_伏笔与时间线.md`：时间顺序、伏笔埋设、回收状态和信息差。

-   `02_大纲与设定/05_后续写作风险.md`
    只维护与已发布正文绑定的后续写作风险、承接提醒和逻辑风险，不作为新增正史设定的依据，不替代四份核心文档。

-   发布稿、封面和已经发布的正文副本放在 `03_发布`。

-   已发布正文以 `03_发布` 中的发布稿或用户明确确认的最终发布稿为准；若 `03_发布` 缺少对应章节或与 `01_正文` 不一致，必须先说明差异，不得擅自合并或覆盖。

-   `04_每天`
    用于按日期归档当天写作产出。每天建立一个年月日目录，格式沿用项目现有写法，例如
    `2026.7.6`。

-   `04_每天/日期目录`
    内固定保存当天对应的三类文件：`小说大纲.md`、`小说内容.md`、`小说走向.md`。

-   当天新增或修改的正文、当日大纲整理、后续走向安排，应同步记录到当天日期目录下的对应文件中。

-   正文写作任务默认包含本次目标正文源文件，以及当天日期目录下的 `小说大纲.md`、`小说内容.md`、`小说走向.md`；用户无需在每次写作指令中重复点名这些归档文件。

-   `04_每天`
    只作为每日写作归档和过程记录，不作为人物设定、世界观、全书大纲、章节规划或伏笔时间线的最高依据；若与
    `02_大纲与设定` 四份核心文档冲突，以文件优先级规则处理。

-   不得在其他目录重复维护人物设定、世界观、大纲或伏笔。

-   除 `04_每天`
    的日期目录及其固定三类文件外，不得随意创建新的大纲、人物表、时间线、日报、审计或优化报告文件。

-   不得擅自删除、移动、改名或合并现有文件。

-   除正文写作任务默认同步的每日归档文件外，每次只修改用户明确指定或当前任务直接要求的文件。

-   不得以统一风格、优化结构或保持一致为理由，顺便修改其他文件。

-   除非用户明确点名要求修改
    `AGENTS.md`，否则不得擅自修改、重写、移动、删除或覆盖 `AGENTS.md`。

-   大范围修改指多章正文、核心设定、章节规划、伏笔时间线或风险文档的批量改动。

-   大范围修改优先使用 Git 保存修改前状态；已有 Git 时先查看工作区状态，不得回退、覆盖或丢弃用户已有改动；没有 Git
    时，必须先获得用户确认。

## 八、已发布章节保护

-   第001章至第010章已经发布。
-   未经用户明确点名授权，不得修改、重写、改名、移动、删除或覆盖第001章至第010章。
-   后续剧情必须兼容第001章至第010章已经确定的内容。
-   用户明确告知新增已发布章节时，即使尚未更新本文件，也必须先按最新告知内容保护对应章节；若用户要求更新 `AGENTS.md`，同步更新本节保护线。
-   已发布章节保护线更新后，必须检查新增已发布章节是否给 `02_大纲与设定/05_后续写作风险.md` 带来新的绑定风险；需要补充风险时先说明范围，再按用户指令更新。
-   如果总体大纲与已发布正文冲突，以已发布正文为准，并在任务结果中指出冲突。

## 九、文件优先级

发生剧情、人物或时间线冲突时，按照以下顺序处理：

1.  已发布正文。
2.  `02_大纲与设定/01_全书总体大纲.md`。
3.  `02_大纲与设定/02_人物与世界观.md`。
4.  `02_大纲与设定/03_分卷与章节规划.md`。
5.  `02_大纲与设定/04_伏笔与时间线.md`。
6.  尚未发布的正文。

不得静默修改高优先级文件来适配低优先级文件。

## 十、完成后的检查

完成正文或剧情修改后，必须检查：

-   男主是否始终为凉生；
-   女主是否始终为露雨；
-   是否违反单女主和无后宫规则；
-   人物掌握的信息是否合理；
-   时间线是否连续；
-   是否擅自增加新规则、新能力或万能道具；
-   是否出现前后文开篇和片尾重复剧情，或连续章节机械重复同一种收尾方式；
-   是否与已发布正文冲突；
-   是否遗漏本章应推进或回收的伏笔；
-   是否遗漏、误解或错误解决与本章相关的后续写作风险；
-   若本章将写作风险转化为伏笔或长期线索，是否仍兼容总体框架且不存在逻辑问题；
-   是否保持紧张、悬疑、诡异和职场规则怪谈主调；
-   幽默是否自然且没有破坏危险感；
-   是否存在模板句、机械排比、重复解释、过度总结等明显 AI 痕迹；
-   复杂逻辑是否优先通过凉生的观察、验证和解释呈现；
-   思维导图是否确有必要、足够简洁且没有代替正文推理；
-   爽点是否来自已有规则、信息差、铺垫和人物能力；
-   是否出现无铺垫开挂、人物降智或廉价打脸式爽点；
-   是否修改了任务范围之外的文件。
